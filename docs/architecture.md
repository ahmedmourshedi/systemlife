# معمارية النظام

## الهدف

النظام مصمم ليكون مركزًا عربيًا موحدًا لإدارة الحياة الشخصية، مع فصل واضح بين:

- الواجهة.
- المصادقة.
- قاعدة البيانات.
- الحماية.
- مركز اليوم للحساب الشخصي.
- صفحات صيانة النظام.

## الطبقات

### 1. طبقة الواجهة

موجودة داخل:

```text
app/
components/
```

وتعتمد على:

- Next.js App Router.
- Server Components للقراءة.
- Server Actions للإدخال والتعديل.
- Tailwind CSS للتصميم.

### 2. طبقة المصادقة

موجودة في:

```text
lib/supabase/
proxy.ts
app/auth/callback/route.ts
```

وتستخدم Supabase Auth مع جلسات Cookies حتى تعمل مع SSR.

### 3. طبقة البيانات

موجودة في:

```text
lib/data.ts
app/actions.ts
database/schema.sql
```

القراءة تتم من `lib/data.ts`، والكتابة تتم من `app/actions.ts`.

### 4. طبقة الحماية

الحماية الأساسية في قاعدة البيانات عبر RLS:

- كل جدول شخصي يحتوي `user_id`.
- المستخدم يرى سجلاته فقط.
- الأدمن يرى كل السجلات.
- جدول `profiles` يتحكم في صلاحية `admin` و`user`.

### 5. طبقة PWA

موجودة في:

```text
app/manifest.ts
public/sw.js
components/register-service-worker.tsx
```

وتوفر:

- أيقونة تطبيق.
- فتح كتطبيق مستقل.
- صفحة Offline أساسية.
- كاش للملفات الأساسية.

## لماذا هذه المعمارية؟

لأن النظام شخصي لكنه قابل للتوسع. يمكن استخدامه لمستخدم واحد فقط، أو توسيعه لاحقًا لعدة مستخدمين مع إدارة أدمن.
